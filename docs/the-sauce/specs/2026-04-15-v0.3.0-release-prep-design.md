# v0.3.0 Release Prep — Design Spec

**Date:** 2026-04-15
**Branch:** claude/init-repo-DpH9p
**Target release:** v0.3.0 (pilot stage, sideload distribution)

## Context

AI Toolkit is currently at v0.2.1 with an uncommitted feature in the working tree that adds **plugin sideload + flat-layout toolkit support**. Two internal review passes have landed (2026-04-05 and 2026-04-10). A fresh full code review surfaced:

- 9 blockers (7 after Q5 scoping — see below)
- 10 High findings
- 11 Medium findings
- 10 Low/nit findings
- Zero unit tests for four critical modules: `extension.ts`, `copilotSettings.ts`, `treeProvider.ts`, `dashboard.ts`

The goal of this spec is to land all Blockers + High + Medium findings, establish baseline test coverage, ship the uncommitted sideload feature correctly, and tag v0.3.0 as a polished pilot-stage release distributed via GitHub release `.vsix` (not Marketplace).

## Goals

1. Polished, pilot-stage v0.3.0 release distributed as a tagged GitHub `.vsix`.
2. Land the dirty-tree sideload/flat-layout feature, with its review findings fixed.
3. Clean up all effective Blockers, all High, all Medium findings.
4. Establish baseline happy-path test coverage on the four untested modules.
5. No external network calls from the dashboard.
6. `.vsix` package free of source maps, declaration files, orphan artifacts, and dev-only scripts.

## Non-Goals

- Public VS Code Marketplace publication (sideload-only distribution per user decision).
- Marketplace polish fields: PNG icon, galleryBanner, badges, `homepage`, `bugs`, `qna`, expanded `keywords`, bumped `engines.vscode` minimum.
- Removing `preview: true` from the manifest — stays per pilot-stage status.
- Low-priority nits L1–L4, L9, L10 — defer to 0.3.1.
- Dashboard refactor into separate static files (M7) — too big.
- `ExtensionContext` class refactor (M9) — too big.
- `aiToolkit.logLevel` / `aiToolkit.maxScanDepth` settings — document only, no new settings.
- Net-new features beyond the dirty-tree sideload work.
- 100% test coverage — happy-path baseline only.

## Approach — Theme-Based Sequential Commits

Single release branch, five themed commit groups landing in order. Each theme is independently verifiable (compile + test + smoke) and reads well as a commit. The sideload dirty-tree feature lands inside theme 2 where its fixes belong.

Alternatives considered and rejected:

- **Linear severity sweep** (Blockers → High → Medium → tests). Rejected: huge long-lived diff, worse to review, mixes concerns across commits.
- **Parallel worktrees by theme**. Rejected: worktree juggling overhead doesn't pay back for a solo developer on a cleanup task.

## Testing Strategy

- **Unit tests:** node:test + node:assert/strict, no mocking framework. Pattern follows existing test files. Fake dependencies via plain object literals.
- **No `@vscode/test-electron`.** Anything that genuinely requires the Extension Host goes on T5.3's manual smoke checklist.
- **Regression guards:** existing 117 tests must pass at every theme boundary. Run `npm run check` after each theme before committing.
- **Manual smoke (T5.3):** final gate before tag. Any failure triggers a full re-run after fixes.

What we are **not** testing automatically:

- VS Code command execution (Extension Host territory)
- Tree view visual rendering
- Webview visual interaction
- Real git network operations
- Settings sync across real VS Code installs

These are covered by manual smoke only.

## Risks & Mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| T2.3 folder-children restoration reintroduces the UX the dirty tree tried to escape | Medium | Before implementing, re-read the dirty diff's rationale. If there was a real reason, fall back to "click opens SKILL.md". |
| T4.1 `extension.ts` testability refactor grows out of control | Medium | Hard cap: if a test needs > ~20 lines of production change, skip it and defer. |
| T1.1 removing sourcemaps breaks F5 debugging DX | Low | Breakpoints still work in `.js`. If painful, revert `sourceMap: false`; keep the rest. |
| T1.8 dashboard font change looks visibly worse | Low | `var(--vscode-font-family)` matches native UI; accept. |
| Theme 3 edits conflict with Theme 2 edits | Medium | Theme 2 lands before Theme 3. Theme 3 starts from Theme 2's HEAD. |
| Manual smoke test surfaces a late bug | Low–Medium | Budgeted into T5.3, not a separate task. |
